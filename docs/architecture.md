# Архитектура

Эта страница — для тех, кто хочет заглянуть внутрь: разработчиков и тех, кто думает
дополнять бенчмарк. Нужна просто методика — она на странице
[Как это работает](how-it-works.md).

!!! note "Главный принцип одной фразой"
    **Код читает данные, данные не живут в коде.** Все числа, пороги и правила лежат в
    отдельных файлах-настройках, а программа их оттуда читает — не зашивает в себя.
    Поменять метрику = поправить файл, а не лезть в код.

## Зачем так

Бенчмарк имеет смысл, только если его результат **воспроизводим**: тот же код, та же
задача, та же версия правил → та же оценка. Поэтому правила вынесены из кода в файлы:

- одно место правды для каждой настройки (нет вторых копий, которые разъезжаются);
- видно, что именно поменялось между версиями (поправили файл — это и есть изменение метрики);
- проверяющий код не «знает» порогов — он их читает, поэтому не может тихо разойтись с
  тем, что написано в правилах.

## Две половины: данные и код

Репозиторий делится на **данные** (правда о том, что и как мы меряем) и **код**
(инструмент, который это исполняет).

**Данные — что и как оценивать:**

- `metrics/` — правила метрики SMOP: общие определения осей + как их считает машина и
  как ставит эксперт.
- `tasks/` — сами задания: условие, скрытые тесты, эталонное решение (а для платформенных
  задач ещё и описание учебной базы).
- `editions/` — профили прогона: какие оси считать и в каких условиях (об изданиях ниже).
- `generation/` — с какими моделями работаем: список моделей, их параметры, системные
  промпты.

**Код — чем оценивать (папка `harness/`):**

- `generate/` — попросить модели написать код по задачам (есть и офлайновая имитация
  `--mock` — прогнать конвейер без сети);
- `score/` — оценить полученный код по осям S · M · O · P;
- `execute/` — запустить код: OneScript для категории A, 1С без интерфейса (в Docker) для
  категории B;
- `stats/` — посчитать согласие машины с экспертом;
- `preflight` — предполётные проверки готовности (`doctor` — инструменты и ключи; `ping` —
  живая связь с моделями); `submit` — упаковка прогона для шеринга с отпечатком версии
  бенчмарка (`compat_hash`).

## Поток данных: от задачи до оценки

```
задания + список моделей + профиль прогона (издание)
        │
        ▼  generate — модель пишет код
        │  (для категории B сначала автоматически собираются сведения о базе)
   код-кандидаты  →  сохраняются в results/
        │
        ▼  score — код ЗАПУСКАЕТСЯ: компилятор · тесты · реальная 1С
   оценка по четырём осям  [S, M, O, P]  +  общий балл Q
        │
        ├──▶  лидерборд: рейтинг моделей (Уровень 1, машина)
        │         └──▶  submit — пакет с отпечатком версии бенчмарка (шеринг результата)
        ▼  эксперт сверяется с машиной на части результатов (Уровень 2)
   знак «проверено» (Verified) + согласие машины и эксперта
```

Подробнее, что означает каждая ось и зачем две линейки оценки, — на странице
[Как это работает](how-it-works.md).

## Издания — «в каких условиях испытывали»

**Издание** — это профиль прогона: какие оси считаем и при каких условиях модель решала
задачу. Один и тот же набор задач можно прогнать по-разному:

- **core** (есть сейчас) — модель получает задание и пишет код; для платформенных задач
  она может сама собрать нужные сведения о базе (метаданные);
- **agent** (в планах) — полный агентный цикл: модель не просто пишет код, а сама правит
  конфигурацию инструментами и проверяет результат.

Издание отвечает на вопрос «*в каких условиях* испытывали», а не «*кто проверил*
результат». «Проверено экспертом» (Verified) — это отдельное измерение, не издание.

## Где исполняется код: local и docker

Код кандидата нужно запустить, и есть два режима:

- **docker** (по умолчанию) — код запускается в изолированном контейнере: без доступа к
  сети, с лимитами по памяти и времени, файлы только для чтения. Это дефолт, потому что код
  от нейросети недоверенный — ему не место на хосте без песочницы. Так же гоняется и в CI.
- **local** — инструменты стоят на твоём компьютере. Быстрее, удобно для своей разработки;
  включается явно (`--runner local` / `--bsl local` или `PRISM_RUNNER=local`).

Важно: **способ запуска на оценку не влияет.** Инструмент той же версии, балл тот же.
Поэтому режим — это инфраструктура, а не часть результата (он не входит в «версия × издание
× конфиг»). Выбор — через `PRISM_RUNNER` / `PRISM_BSL` (по умолчанию `docker`).

## Принципы, которые держат бенчмарк честным

- **Код читает данные.** Пороги и правила — в файлах метрики, не в коде.
- **Модель — это факты, канал доступа — отдельно.** Сама модель описана фактами (кто
  сделал, что умеет), а как до неё достучаться — отдельной настройкой. Добавить нового
  провайдера = один файл-адаптер, остальной код не трогаем.
- **Задача = условие + скрытая проверка**, лежат рядом.
- **Операционная надёжность отдельно от метрики.** Сохранение прогресса, повторы при
  сбоях сети, учёт стоимости, параллельный запуск — всё это нужно для больших прогонов,
  но на баллы SMOP **не влияет**: балл считается из ответа модели, а не из того, как и
  почём он получен. Параллельный скоринг (`PRISM_CONCURRENCY`) проверен бит-в-бит против
  последовательного — баллы совпадают.
