# Архитектура PRISM

Зафиксировано: 2026-06-12. Это прозаическое описание — числа и правила живут в YAML
(`metrics/`, `editions/`, `generation/`), здесь только ссылки на них, не копии.

## Решение: один репозиторий бенчмарка

**PRISM поглощает функции `genlab-1c-core` и становится единственным живым репозиторием
бенчмарка** (задачи + метрика + издания + харнесс). Причины:

1. **Один источник правды.** core нёс вторые копии метрики, задач и каталога моделей
   (`configs/`) — два репозитория требовали вечной синхронизации.
2. **Версионирование результата.** `результат = версия × издание × конфиг` честен только
   когда версия одного репозитория фиксирует и задачи, и метрику, и скореры.
3. **Скореры читают метрику.** Пороги и банды L1 живут в `metrics/smop_l1_auto.yaml`,
   код их читает, а не хардкодит. Разнести код и метрику по репозиториям = разорвать правду.
4. **У core нет независимого потребителя.** web вендорил его ради статистики и клиентов —
   переключение submodule на prism решает это.

Судьба остальных репозиториев:

| Репозиторий | Статус | Роль |
|---|---|---|
| **prism** | живой | бенчмарк целиком: актив + Уровень 1 (авто-оценка) |
| **expert-platform** (genlab-1c-web) | живой | Уровень 2: коллективная экспертная разметка. Свой стек (FastAPI+React+PostgreSQL), своя лицензия, свой деплой. Подключает prism как submodule |
| **core-bench** (genlab-1c-core) | заморожен | артефакт НИР-2026: воспроизводимость мартовского эксперимента, статья ссылается на него |
| **prism-legacy** | заморожен | исходный концепт SMOP (март 2026) |

## Структура репозитория

```
prism/
│  ДАННЫЕ (источник правды)
├─ metrics/            конституция SMOP + протоколы L1 (авто) / L2 (эксперт)
├─ tasks/              задачи: category_a/<id>/{task,tests,canonical}, category_b/…
├─ editions/           конфиг-профили прогона (core; позже ctx)
├─ generation/         models (каталог) / params (числа) / prompts (system-тексты)
│
│  КОД (харнесс; переезжает из core-bench и prism-concept)
├─ harness/
│  ├─ run.py           CLI: prism run --edition core --model <ключ>
│  ├─ generate/        батч-генерация
│  │  ├─ adapters/     каналы доступа к LLM: openrouter, openai_compat (Ollama/vLLM),
│  │  │                gigachat, yandexgpt, … — один контракт generate() на всех
│  │  └─ context/      MCP-загрузчик метаданных (категория B, context: mcp)
│  ├─ score/           L1-скореры S/M/O/P — пороги читают из metrics/smop_l1_auto.yaml
│  ├─ execute/         песочницы: onescript/ (кат. A), onec/ (кат. B, синтетическая конфа)
│  ├─ stats/           согласие L1↔L2: каппа Коэна, доверительные интервалы
│  └─ report/          отчёты HTML/LaTeX/JSON, графики
│
│  ВЫХОДЫ
├─ results/            сырые результаты прогонов (хешируются)
├─ leaderboard/        агрегаты + статический сайт (позже)
│
│  ОБВЯЗКА
├─ tools/              бинари: BSL LS, OneScript, YAXUNIT (gitignored)
├─ docker/             образы песочниц
└─ docs/               проза (этот файл, методика) — ссылается на YAML, не дублирует
```

## Поток данных (прогон издания core)

```
editions/core.yaml + generation/* + tasks/**
        │
        ▼  harness/generate  ── адаптер из models.yaml (access.adapter);
        │                       категория B: MCP → синтетическая база метаданных
   код кандидатов → results/ (с хешами)
        │
        ▼  harness/score     ── банды из metrics/smop_l1_auto.yaml;
        │                       исполнение: execute/onescript (A) / execute/onec (B)
   L1-оценки: вектор [S,M,O,P] + Q
        │
        ├──▶ leaderboard / report
        │
        ▼  экспорт прогонов в expert-platform
   L2-оценки экспертов (вслепую к L1, по metrics/smop_l2_expert.yaml)
        │
        ▼  harness/stats
   согласие L1↔L2 (κ) — публикуемый научный результат
```

## Режимы исполнения (local | docker)

Исполнение кода кандидатов идёт через раннер (`harness/execute/runner.py`), скореры
о режимах не знают. Режим — **инфраструктура, не идентичность результата**: баллы не
зависят от способа запуска (та же пиновая версия инструмента), поэтому режим не входит
в «версия × издание × конфиг» и не живёт в `editions/`. Выбор — env `PRISM_RUNNER`
(по умолчанию `local`) или аргументом.

| Режим | Что | Когда |
|---|---|---|
| `local` | инструмент из `tools/` на хосте | своя разработка (быстро) |
| `docker` | образ `prism-onescript` (`docker/onescript.Dockerfile`): без сети, лимиты CPU/память/pids, код смонтирован read-only, непривилегированный пользователь | CI и недоверенные кандидаты — код LLM произволен, у OneScript есть доступ к ФС/сети |

Бинари инструментов в git не идут (`tools/` в .gitignore) — коммитятся только
bootstrap-скрипты (`tools/get-*.sh`) с пиновыми версиями; docker-образы собираются
из тех же официальных релизов.

## Принципы

- **Код читает данные, данные не живут в коде.** Скорер не знает порогов (берёт из
  `metrics/`), раннер не знает осей (берёт `scorers` издания и `applies_to` конституции),
  генератор не знает протокола доступа (берёт `access.adapter` из каталога моделей).
- **Вендор ≠ канал доступа.** Модель описывается фактами (vendor, capabilities), канал —
  адаптером (`access`). Добавить провайдера = один файл-адаптер + запись в каталоге.
- **Задача = условие + проверка.** prompt живёт рядом со скрытыми тестами в `tasks/<id>/`;
  параметры генерации — отдельно (`generation/`).
- **Два уровня оценки.** L1 (машина) строит лидерборд; L2 (эксперт) калибрует L1 и даёт
  Verified-подмножество. Протоколы разные, конституция общая — см. `metrics/smop.yaml`.
  Границы конструкта, где объективность ещё не заработана (категория B, κ на одном
  эксперте, censored-шкала O), и приоритеты ремонта — `docs/validity.md`.

## Порядок переноса (ничего не ломаем)

1. Скореры категории A (из prism-concept) + минимальный `harness/run.py` — без core.
2. Генерация: адаптеры (первый — openrouter из core) + MCP-загрузчик.
3. Статистика и отчёты из core → согласие L1↔L2.
4. expert-platform переключает submodule core → prism; core-bench замораживается.
   До этого момента core-bench живёт нетронутым.
