# Как запустить

Вся работа с бенчмарком идёт через одну команду — `prism`. На этой странице — установка
и три команды, которые покрывают весь цикл.

!!! note "Три команды, и всё"
    | Команда | Что делает |
    |---------|------------|
    | `prism generate` | попросить модели написать код по задачам |
    | `prism check` | проверить, что бенчмарк цел: задания, эталоны, инструменты |
    | `prism score` | оценить готовый код по осям S·M·O·P |

## Установка

```bash
python -m venv .venv && source .venv/bin/activate
pip install -e ".[dev]"          # харнесс: PyYAML + pydantic, dev — pytest

./tools/get-onescript.sh         # ось M (кат. A): исполнение тестов в OneScript
./tools/get-bsl-ls.sh            # оси S/O: статический анализатор BSL LS (нужна java 21+)

# оси M/P категории B: учебная 1С headless в Docker (дистрибутив 1С приносишь сам,
# в репозиторий и образы он не попадает — см. docker/onec.Dockerfile):
docker build -f docker/onec.Dockerfile -t prism-onec tools/1ce-training
```

После `pip install -e .` доступна команда `prism`; без установки — то же самое через
`python3 -m harness.cli <команда>`.

## `prism generate`

Генерация кода кандидатов моделями по изданию: для каждой пары (задача, модель)
собирается контекст (кат. B — агентным сбором метаданных), делается N прогонов с seed,
ответы хешируются (детерминизм) и пишутся в `results/experiment_*.json` — в той же схеме,
которую читает `prism score`.

```bash
prism generate --category A                          # все модели, все задачи A
prism generate --category B --models claude gpt      # выбор моделей
prism generate --category A --tasks A1 A2            # выбор задач
```

| Флаг | По умолчанию | Назначение |
|------|--------------|------------|
| `--category` | — (обязателен) | `A` или `B` |
| `--edition` | `core` | издание из `editions/` |
| `--models` | все из каталога | ключи моделей (`generation/models.yaml`) |
| `--tasks` | все категории | id задач |
| `--resume EXP_NAME` | — | дозапустить эксперимент: пропустить готовые пары |
| `--concurrency N` | из `params.yaml` (4) | параллельных пар задача×модель |
| `--max-cost USD` | без капа | новые пары не запускать сверх порога стоимости |
| `--retries N` | `3` | повторов при транзиентном сбое сети |
| `--dry-run` | — | только предполётная оценка стоимости, без вызовов сети |

Нужны ключи API в окружении (см. `harness/generate/adapters/registry.py`).

### Операционная надёжность (прогон на сотнях задач)

Слой поверх метрики — на баллы SMOP не влияет.

- **Чекпойнт на пару.** Каждый готовый результат сразу пишется в
  `results/<exp>.parts/<task>__<model>.json` (атомарно). Падение на 60/100 → готовое
  не теряется; финал собирается из частей.
- **Resume.** `--resume <exp_name>` грузит завершённые части и дозапускает только
  недостающие/упавшие пары. Сеть умерла — добиваем остаток, а не гоним всё заново.
- **Ретраи.** Транзиентный сбой (`429`/`5xx`/таймаут/обрыв) повторяется с экспоненциальной
  паузой; перманентный (`401`/нет кредитов) — сразу, повтор бессмыслен.
- **Параллелизм.** Независимые пары идут пулом потоков (`--concurrency`); опциональный
  кап на канал доступа (`provider_concurrency` в `params.yaml`) — для провайдеров с
  жёсткими лимитами. Порядок результатов детерминирован независимо от порядка завершения.
- **Бюджет.** Живой счётчик стоимости, мягкий кап `--max-cost`, предполётная оценка
  `--dry-run`. Цены — в датированной таблице `generation/pricing.yaml` (волатильны,
  поэтому отдельно от каталога моделей); модель без цены считается нулевой с
  предупреждением (бюджет недосчитан, а не врёт).

## `prism check`

Проверка целостности — «прехук» данных бенчмарка. Запускается без аргументов:

```bash
prism check
```

Четыре среза, от дешёвого к дорогому:

1. **Контракты метрики** — конституция/протокол/издания/генерация грузятся и
   согласованы (оси совпадают, банды ведут в достижимые значения, веса O положительны,
   скореры изданий известны, параметры покрывают каталог моделей).
2. **Контракты заданий** — каждая задача валидна; у исполнимых есть тест-кейсы и
   паттерны entry point. Задачи без тестов помечаются `pending` (предупреждение, не ошибка).
3. **Когерентность эталонов** — сильнейший инвариант: `canonical.bsl` каждой задачи
   проходит свои же скрытые тесты на 100%. Валидирует тесты и эталон разом. Требует
   OneScript; нет инструмента → срез пропускается.
4. **Инструменты по осям** — доступность OneScript (M), BSL LS + java (S/O).

Код возврата: `0` — нарушений нет; `1` — есть хоть один `fail` (нарушенный контракт
или эталон, не прошедший свои тесты). Подходит для CI-гейта.

## `prism score`

Авто-оценка Уровня 1 готовых генераций по изданию — **без новой генерации**. Берёт
выходы моделей из `results/experiment_*.json`, прогоняет скореры осей, считает Q и
пишет результат в схеме, зеркальной экспертной разметке.

```bash
prism score                                          # свежайший experiment_A_*, издание core
prism score --edition core --experiment results/experiment_A_<timestamp>.json
prism score --out /tmp/my_eval.json
```

| Флаг | По умолчанию | Назначение |
|------|--------------|------------|
| `--experiment` | свежайший `experiment_A_*.json` | какой набор генераций оценивать |
| `--edition` | `core` | издание из `editions/` (какие оси считать) |
| `--out` | `results/auto/<exp>_auto_l1.json` | куда писать результат |

Вывод — таблица `S·M·O·P·Q` по каждому кандидату; если рядом лежит экспертный файл
(`results/evaluations/<exp>_expert_*.json`), печатается дельта авто↔эксперт по S/M/Q.

!!! tip "Режим исполнения"
    Раннер оси M выбирается через `PRISM_RUNNER` (`local` по умолчанию | `docker`),
    путь к Java для S/O — через `PRISM_JAVA`. Это инфраструктура, не идентичность
    балла, поэтому не в `editions/`.
