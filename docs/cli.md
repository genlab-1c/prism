# Как запустить

Вся работа с бенчмарком идёт через одну команду — `prism`. На этой странице — установка
и команды, которые покрывают весь цикл.

!!! note "Команды"
    | Команда | Что делает |
    |---------|------------|
    | `prism doctor` | быстрый чек готовности: инструменты осей и ключи моделей (без сети) |
    | `prism ping` | живой минимальный запрос к моделям — проверить ключ и связь |
    | `prism generate` | попросить модели написать код по задачам (`--mock` — сухой прогон) |
    | `prism check` | проверить, что бенчмарк цел: задания, эталоны, инструменты |
    | `prism score` | пересчитать оценку готового кода по осям S·M·O·P |
    | `prism leaderboard` | **мгновенно** показать лидерборд из готовой оценки (без пересчёта) |
    | `prism submit` | упаковать прогон для шеринга (хеш совместимости) или принять чужой |

!!! tip "Первый запуск — по шагам"
    `prism doctor` (всё ли установлено) → `prism ping` (отвечают ли модели) →
    `prism generate --category A --mock` (сухой прогон конвейера без сети) →
    `prism score` (оценить) → `prism submit` (поделиться результатом).

!!! tip "Параллелизм и песочницы — env или флаг"
    Долгие операции (`score`, `check`) идут параллельно; ручки управления едины — env
    **или** одноимённый флаг CLI (флаг приоритетнее): `PRISM_CONCURRENCY`/`--concurrency`,
    `PRISM_RUNNER`/`--runner`, `PRISM_BSL`/`--bsl`. В интерактивном терминале виден
    прогресс-бар; в пайпе/CI он гаснет сам.

## Установка

Окружение управляет [uv](https://docs.astral.sh/uv/) — единый менеджер Python/зависимостей.
Нет uv? Поставьте одной строкой: `curl -LsSf https://astral.sh/uv/install.sh | sh`.

По умолчанию код кандидата исполняется **в Docker** (песочница для недоверенного кода ИИ),
поэтому базовый путь — собрать образы инструментов:

```bash
uv sync                          # харнесс + dev-группа (pytest, ruff…) в .venv (uv ведёт сам)
make images                      # docker-образы инструментов A: prism-onescript + prism-bsl-ls
make image-onec                  # учебная 1С для категории B (подскажет про дистрибутив)
# короче: make setup-all = uv sync + образы A + учебная 1С B
```

Хостовый dev-режим (быстрее, но без песочницы) — `./tools/get-onescript.sh` +
`./tools/get-bsl-ls.sh` (нужна java 21+), и звать с `--runner local` / `--bsl local`.

Команды запускаются через `uv run` (он сам поддерживает окружение): `uv run prism <команда>`.
То же без установки пакета — `uv run python -m harness.cli <команда>`.

**Одной целью:**

| Цель | Что ставит |
|------|------------|
| `make setup-all` | **всё в Docker (дефолт)**: окружение + образы инструментов A + учебная 1С B |
| `make setup-docker` | окружение + docker-образы инструментов A (без учебной 1С B) |
| `make setup` | хостовый dev-режим: инструменты осей на хост (зови с `--runner/--bsl local`) |
| `make image-onec` | образ учебной 1С для категории B (проведёт по дистрибутиву, если его нет) |

Категория B требует **свой** дистрибутив платформы 1С (не редистрибутируется): `make
setup-all` поставит всё для A и подскажет шаги для B (скачать учебную версию с
developer.1c.ru → положить `.run` в `tools/1ce-training/` → `make image-onec`).
Проверить, что всё на месте, — `prism doctor`.

## `prism doctor`

Быстрый (секунды, **без** прогонов в 1С и сетевых вызовов) ответ на вопрос «всё ли у
меня есть, чтобы работать». Три среза: окружение (Python/uv), инструменты осей (OneScript,
BSL LS+java, Docker-образ `prism-onec`) и какие ключи моделей заданы в `.env` — по каналам
доступа. В конце — вердикт «готов к: генерации · оценке A · оценке B».

```bash
prism doctor
```

Каждый недостающий пункт несёт подсказку, что доставить (`make images`, вписать ключ в
`.env` и т.п.). Сети не трогает — за проверкой, что ключ рабочий, идите в `prism ping`.

## `prism ping`

Живой минимальный запрос к каждой модели каталога: проверяет, что ключ канала **рабочий**
и провайдер отвечает. ⚠️ **Тратит немного токенов** — по одному короткому запросу на модель.
Модель без ключа не ошибка, а пропуск (запрос не делается).

```bash
prism ping                          # все модели с заданными ключами
prism ping --models deepseek gemini # только выбранные
```

Код возврата: `1`, если хоть одна модель **с ключом** не ответила (годится для CI);
пропуски на код не влияют.

## `prism generate`

Генерация кода кандидатов моделями по изданию: для каждой пары (задача, модель)
собирается контекст (кат. B — агентным сбором метаданных), делается N прогонов с seed,
ответы хешируются (детерминизм) и пишутся в `results/experiment_*.json` — в той же схеме,
которую читает `prism score`.

```bash
prism generate --category A                          # все модели, все задачи A
prism generate --category B --models claude gpt      # выбор моделей
prism generate --category A --tasks A1 A2            # выбор задач
```

| Флаг | По умолчанию | Назначение |
|------|--------------|------------|
| `--category` | — (обязателен) | `A` или `B` |
| `--edition` | `core` | издание из `editions/` |
| `--models` | все из каталога | ключи моделей (`generation/models.yaml`) |
| `--tasks` | все категории | id задач |
| `--resume EXP_NAME` | — | дозапустить эксперимент: пропустить готовые пары |
| `--concurrency N` | из `params.yaml` (4) | параллельных пар задача×модель |
| `--max-cost USD` | без капа | новые пары не запускать сверх порога стоимости |
| `--retries N` | `3` | повторов при транзиентном сбое сети |
| `--dry-run` | — | только предполётная оценка стоимости, без вызовов сети |
| `--mock [canonical\|stub]` | — | сухой прогон конвейера без сети: имитация модели `mock/echo` |

**Сухой прогон (`--mock`).** Чтобы прогнать весь путь генерация→оценка→лидерборд, не
тратя ключей и денег: вместо реальной модели подставляется офлайновая `mock/echo`, которая
отдаёт эталон задачи (`--mock` или `--mock canonical`) либо заглушку (`--mock stub`). На
выходе настоящий `experiment_*.json` — дальше его скармливают `prism score`. Удобно
убедиться, что окружение оценки собрано, до первого платного прогона.

Нужны ключи API: задайте их в окружении или в `.env` (копия `.env.example`) — он
подхватывается автоматически (pydantic-settings, см. `harness/settings.py`). Какой ключ
нужен для модели — по `access.adapter` в `generation/models.yaml`.

### Операционная надёжность (прогон на сотнях задач)

Слой поверх метрики — на баллы SMOP не влияет.

- **Чекпойнт на пару.** Каждый готовый результат сразу пишется в
  `results/<exp>.parts/<task>__<model>.json` (атомарно). Падение на 60/100 → готовое
  не теряется; финал собирается из частей.
- **Resume.** `--resume <exp_name>` грузит завершённые части и дозапускает только
  недостающие/упавшие пары. Сеть умерла — добиваем остаток, а не гоним всё заново.
- **Ретраи.** Транзиентный сбой (`429`/`5xx`/таймаут/обрыв) повторяется с экспоненциальной
  паузой; перманентный (`401`/нет кредитов) — сразу, повтор бессмыслен.
- **Параллелизм.** Независимые пары идут пулом потоков (`--concurrency`); опциональный
  кап на канал доступа (`provider_concurrency` в `params.yaml`) — для провайдеров с
  жёсткими лимитами. Порядок результатов детерминирован независимо от порядка завершения.
- **Бюджет.** Живой счётчик стоимости, мягкий кап `--max-cost`, предполётная оценка
  `--dry-run`. Цены — в датированной таблице `generation/pricing.yaml` (волатильны,
  поэтому отдельно от каталога моделей); модель без цены считается нулевой с
  предупреждением (бюджет недосчитан, а не врёт).

## `prism check`

Проверка целостности — «прехук» данных бенчмарка. Запускается без аргументов:

```bash
prism check
```

Четыре среза, от дешёвого к дорогому:

1. **Контракты метрики** — конституция/протокол/издания/генерация грузятся и
   согласованы (оси совпадают, банды ведут в достижимые значения, веса O положительны,
   скореры изданий известны, параметры покрывают каталог моделей).
2. **Контракты заданий** — каждая задача валидна; у исполнимых есть тест-кейсы и
   паттерны entry point. Задачи без тестов помечаются `pending` (предупреждение, не ошибка).
3. **Когерентность эталонов** — сильнейший инвариант: `canonical.bsl` каждой задачи
   проходит свои же скрытые тесты на 100%. Валидирует тесты и эталон разом. Требует
   OneScript; нет инструмента → срез пропускается. Эталоны гоняются **параллельно**
   (кат. B — с прогресс-баром); сузить — `--task B17` / `--category B`.
4. **Инструменты по осям** — доступность OneScript (M), BSL LS + java (S/O),
   Docker-образ `prism-onec` (S/M/P кат. B). Тот же срез без прогона эталонов — `prism doctor`.

Параллелизм и песочницы — флагами `--concurrency` / `--runner` / `--bsl` (поверх env),
как у `score`. Код возврата: `0` — нарушений нет; `1` — есть хоть один `fail` (нарушенный
контракт или эталон, не прошедший свои тесты). Подходит для CI-гейта.

## `prism score`

Авто-оценка Уровня 1 готовых генераций по изданию — **без новой генерации**, но **с
пересчётом** (исполнение кандидатов в песочнице: кат. A — OneScript, кат. B — 1С, по
умолчанию обе в Docker, отсюда минуты). Берёт выходы моделей из `results/experiment_*.json`,
прогоняет скореры осей,
считает Q и пишет результат в `results/auto/<exp>_auto_l1.json` (схема зеркальна
экспертной разметке). Во время прогона виден прогресс-бар.

```bash
prism score                                          # свежий прогон A и B → два лидерборда
prism score --experiment results/experiment_A_<timestamp>.json
prism score --full                                   # построчная таблица S·M·O·P·Q вместо лидерборда
prism score --concurrency 8 --runner local           # хостовый раннер флагом (по умолчанию docker)
```

Без `--experiment` оценивается свежий прогон **каждой** категории (A и B), каждый в свой
`auto_l1` — категория B больше не выпадает. `--out` имеет смысл только с явным `--experiment`.

| Флаг | По умолчанию | Назначение |
|------|--------------|------------|
| `--experiment` | свежайший каждой категории (A и B) | какой набор генераций оценивать |
| `--edition` | `core` | издание из `editions/` (какие оси считать) |
| `--out` | `results/auto/<exp>_auto_l1.json` | куда писать результат |
| `--full` | — | построчная таблица `S·M·O·P·Q` (+сверка с экспертом) и срезы по тегам |
| `--concurrency` | env / ≤4 | параллельных прогонов (поверх `PRISM_CONCURRENCY`) |
| `--runner` | env / `docker` | песочница оси M, `docker`\|`local` (поверх `PRISM_RUNNER`) |
| `--bsl` | env / `docker` | инструмент S/O, `docker`\|`local` (поверх `PRISM_BSL`) |

По умолчанию печатается **лидерборд** (средний Q по моделям). Полная построчная таблица
и срезы по тегам — под `--full`; если рядом лежит экспертный файл
(`results/evaluations/<exp>_expert_*.json`), в `--full` печатается дельта авто↔эксперт.

## `prism leaderboard`

**Мгновенная** сводка из уже посчитанной оценки L1 — читает `results/auto/*_auto_l1.json`,
**без пере-исполнения в 1С** (доли секунды против минут у `score`). То, чем быстро глянуть
результат, не пересчитывая.

```bash
prism leaderboard                                    # свежий лидерборд A и B (как в README)
prism leaderboard --experiment results/auto/<exp>_auto_l1.json   # только эта оценка
prism leaderboard --full                             # + построчная таблица и срезы по тегам
```

Без `--experiment` печатается свежайший лидерборд **каждой** категории (A и B) — единым
снимком, как в README, а не одна случайная по времени оценка.

!!! tip "Режим исполнения"
    По умолчанию код кандидата исполняется **в Docker-песочнице** (недоверенный код ИИ).
    Раннер оси M — `PRISM_RUNNER`/`--runner` (`docker` по умолчанию | `local`), инструмент
    S/O — `PRISM_BSL`/`--bsl` (тоже `docker`), путь к Java для `local` — `PRISM_JAVA`. Это
    инфраструктура, не идентичность балла, поэтому не в `editions/`.

## `prism submit`

Поделиться своим прогоном так, чтобы автор бенчмарка мог влить его в публичный лидерборд —
и быть уверенным, что цифры **сравнимы**. Оценки L1 на разных версиях бенчмарка несравнимы
(другая метрика или другой набор задач → другие числа), поэтому к результату прикладывается
`compat_hash` — отпечаток версии (конституция + протокол L1 + набор задач и их условия/проверки).

```bash
# контрибьютор: упаковать свежую оценку L1
prism submit                                 # → results/submissions/<exp>_submission.json
prism submit --experiment results/auto/<exp>_auto_l1.json

# автор: принять чужой пакет
prism submit --verify file_submission.json           # сверить версию бенчмарка
prism submit --verify file_submission.json --apply   # + влить в results/auto/
```

Пакет — самодостаточный JSON (оценки + версии метрики/протокола/раннера + список моделей +
`compat_hash`). Транспорт намеренно простой: файл прикладывают к PR или присылают автору,
никакой серверной инфраструктуры. На приёме `--verify` сверяет отпечаток с текущим
репозиторием: совпал → можно вливать (`--apply` кладёт оценку в `results/auto/`, дальше
лидерборд пересобирает `prism docs`); не совпал → прогон на другой версии, цифры несравнимы
(код возврата `1`).
