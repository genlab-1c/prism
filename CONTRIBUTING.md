# Участие в PRISM

Спасибо, что хотите помочь! PRISM растёт силами сообщества 1С, и самый ценный вклад — это
**новые задачи** и **новые прогоны моделей**. Чем шире банк заданий и чем больше моделей на
нём прогнали, тем точнее и полнее ответ на вопрос «какая нейросеть лучше пишет код на 1С и в
чём именно».

Этот документ — пошаговая карта: с чего начать, как добавить задачу, модель или прислать свой
прогон и как убедиться, что вклад примут. Хотите сначала понять, как устроена оценка, —
загляните в [«Как это работает»](docs/how-it-works.md).

## Три правила, на которых всё держится

Нарушение любого — повод вернуть PR на доработку. В них же и суть проекта:

1. **Все настройки — в файлах, а не в коде.** Пороги, веса и правила метрики лежат только в
   `metrics/*.yaml`; код читает их оттуда. Числа в документации не дублируются — текст
   ссылается на эти файлы.
2. **«Не измерено» — это не ноль.** Если ось нечем проверить (нет нужного инструмента), она
   просто исключается из оценки, а не занижает балл.
3. **Эталон обязан проходить свои тесты на 100%.** Задача без рабочего `canonical.bsl` — это
   не задача, а гипотеза. Это автоматически проверяет `prism check`: пока он не зелёный, задачу
   не примут.

## С чего начать

Окружение управляет [uv](https://docs.astral.sh/uv/) — поставьте его, если ещё нет:
`curl -LsSf https://astral.sh/uv/install.sh | sh`. Два инструмента: **`make`** — *поставить*,
**`prism`** — *пользоваться*. По умолчанию код кандидата исполняется **в Docker** (песочница
для недоверенного кода ИИ).

```bash
git clone https://github.com/genlab-1c/prism && cd prism
make setup-all              # окружение (uv) + docker-образы инструментов A + учебная 1С B
uv run prism doctor         # всё ли установлено: инструменты осей и ключи моделей (без сети)
uv run prism check          # целостность: контракты метрики, задания, эталоны, инструменты
make test                   # тесты харнесса (интеграционные сами пропустятся без инструментов)
```

Команды зовутся через `uv run` — отдельно активировать окружение не нужно. Хотите пощупать
конвейер до первого платного прогона — `uv run prism generate --category A --mock` (сухой
прогон без сети) → `uv run prism score`.

**Быстрый хостовый режим** (без Docker, удобно для задач категории A): `make setup` ставит
OneScript и BSL LS на хост, а проверки гоняйте с флагами `--runner local --bsl local` —
например, `uv run prism check --runner local --bsl local`.

Зелёный `prism check` — окружение готово. Перед коммитом ставится `pre-commit` (хуки уже
описаны в `.pre-commit-config.yaml`):

```bash
uv run pre-commit install
make lint         # ruff + pre-commit, как в CI
```

## Способы участвовать

| Вклад | Порог | Что нужно |
|---|---|---|
| **Задача категории A** (алгоритмика) | низкий | только OneScript — ставится `make setup` |
| **Задача категории B** (платформа) | выше | свой дистрибутив 1С + Docker-образ (см. ниже) |
| **Прогон модели** в лидерборд | низкий | свой API-ключ: `generate → score → submit` |
| **Новая модель** в каталог | низкий | запись в `generation/models.yaml` + ключ |
| **Улучшение харнесса** | средний | Python и зелёный `make test` |
| **Экспертная оценка (L2)** | — | пока в планах (см. [ниже](#уровень-l2--экспертная-оценка-в-планах)) |
| **Баг или идея задачи** | низкий | Issue на GitHub |

Идеи для первых задач и дорожная карта — в [`docs/status.md`](docs/status.md) и в открытых
Issues (ярлык `good first task`).

> **Сначала загляните в [банк задач](tasks/README.md)** — там полный список того, что уже
> включено, с навыками и конструкциями 1С. Это бережёт ваше время: дубликаты не нужны, ценен
> **новый паттерн**, а не вариация существующего. Список собирается из `task.yaml`
> автоматически (`prism tasks`), поэтому всегда актуален — копаться в сырых каталогах не надо.

## Добавить задачу категории A (рекомендуемый вход)

Категория A — алгоритмика на чистом BSL, исполняется в **OneScript** (бесплатно, ставится
`make setup`, платформа 1С не нужна). Самый простой способ войти — примерно 10 минут.

Задача — это каталог `tasks/category_a/<id>_<slug>/` из **четырёх файлов**. Скопируйте
`tasks/category_a/A1_group_table/` как образец и заполните:

**`task.yaml`** — условие:
```yaml
id: A99                     # свободный id (проверьте, что не занят в tasks/category_a/)
name: Краткое имя задачи
category: A
difficulty: easy            # easy | medium | hard
entry_point: ИмяФункции     # функция, которую вызовет харнесс
signature: "Функция ИмяФункции(Аргумент) Экспорт"
prompt: |
  Понятная формулировка для модели: что на входе, что вернуть,
  граничные условия. Без подсказок к реализации.
tags:
  skill: [коллекции, алгоритм]   # только значения из tasks/tags.yaml
```

**`tests.yaml`** — скрытые тест-кейсы (ось M):
```yaml
entry_point_patterns:       # регэкспы поиска функции в коде кандидата
  - "ИмяФунк\\w*"
tests:
  - args: [[3, 1, 2]]       # аргументы вызова
    expected: [1, 2, 3]     # ожидаемый результат (deep-equal)
  - args: [[]]
    expected: []
```
Кладите **граничные случаи** (пусто, один элемент, дубли, отрицательные, переполнение) —
именно на них модели и спотыкаются, ради этого тесты и скрытые.

**`canonical.bsl`** — эталонная реализация. Обязана проходить **все** свои тесты на 100% —
иначе задача не пройдёт `prism check`.

**`perf.yaml`** — нагрузочный профиль (ось O). Без него оптимальность у задачи просто **не
измеряется** — в лидерборде вместо балла будет прочерк, поэтому файл нужен:
```yaml
p_opt: 1.0                  # оптимальный класс роста: 1.0 — линейный, 2.0 — квадратичный
sizes: [200, 400, 800, 1600]  # лесенка размеров входа
gen: |                      # код, генерирующий вход размера {n}
  Массив = Новый Массив;
  Для Сч = 1 По {n} Цикл
      Массив.Добавить(Сч);
  КонецЦикла;
call: '{entry}(Массив);'    # замерочный вызов функции кандидата
```
Решение гоняется на каждом размере, число операций берётся из `oscript -codestat`, а
показатель роста сравнивается с `p_opt`. Прикидывая `p_opt`, отталкивайтесь от эталона: как
быстро растёт работа у *правильного* решения — линейно, как `n·log n`, квадратично.

Проверка вклада — одна команда:
```bash
uv run prism check        # ваш эталон прогоняется против ваших тестов
```
Зелёный `prism check` — задачу можно слать в PR.

> **Теги.** Берите только значения из `tasks/tags.yaml` (словарь закрыт — опечатка ловится
> `prism check`). Нужного навыка нет — предложите его в PR отдельным изменением словаря с
> обоснованием.

## Добавить задачу категории B (платформенная)

Категория B — то, ради чего PRISM существует: код исполняется в **реальной 1С против
синтетической базы**, собираемой из описания задачи. Порог выше — нужен Docker-образ с
платформой.

### Барьер: образ 1С (принесите свой дистрибутив)

Дистрибутив платформы 1С **не распространяется** в репозитории по лицензии. Учебную версию
можно бесплатно получить на [developer.1c.ru](https://developer.1c.ru), затем собрать образ:

```bash
make image-onec        # подскажет, куда положить дистрибутив (.run → tools/1ce-training/),
                       # и соберёт образ prism-onec для исполнения категории B
# или сразу всё для A и B: make setup-all
```

Проверить, что образ на месте, — `uv run prism doctor` (строка «оценке B»).

**Лицензионное правило (жёсткое).** В git идут только **описания и скрипты сборки**, всё
собственного авторства. **Никогда не коммитьте**: дистрибутив платформы 1С, собранные
`.cf`/`.dt`, саму базу. База целиком воссоздаётся из `config_spec.yaml` + `fixtures.yaml`
при каждом прогоне.

### Анатомия B-задачи

Каталог `tasks/category_b/<id>_<slug>/` из **шести файлов**. Образец для копирования —
`tasks/category_b/B1_stock_balances/`:

| Файл | Что описывает |
|---|---|
| `task.yaml` | условие, `entry_point_patterns`, `expected_objects` (объекты метаданных — затравка для оси P), теги |
| `config_spec.yaml` | **синтетическая** конфигурация: справочники, документы, регистры (НЕ копия УТ/БП) |
| `fixtures.yaml` | начальные данные базы (детерминированные) |
| `tests.bsl` | скрытые проверки на BSL, исполняются в 1С |
| `canonical.bsl` | эталон — обязан проходить `tests.bsl` на 100% |
| `perf.yaml` | нагрузочный профиль оси O: чем растим базу (`grow`), лесенка `sizes`, `p_opt` |

Один прогон даёт сразу три сигнала: **M** (вернул правильный ответ), **P** (не упал на
обращении к метаданным) и **O** (как растёт число обращений к СУБД, когда база пухнет).
Без `perf.yaml` оптимальность посчитает разбор кода по типовым антипаттернам — грубее, чем
замер, поэтому профиль лучше написать. Поэтому пишите тесты так, чтобы заведомо сломанный кандидат
(обращение к несуществующему регистру или реквизиту) **проваливался**, а не получал ложный
зачёт. Это типовая ошибка авторов задач — поймайте её, написав нарочно «вредительский» вариант.

### CI не проверяет B автоматически

В CI нет платформы 1С, поэтому канонические B-эталоны там не валидируются. В PR с B-задачей:

1. Прогоните локально `uv run prism check --task <ваша B-задача>`.
2. **Приложите вывод** в описание PR (что эталон прошёл свои тесты).
3. Мейнтейнер с образом перепроверит.

## Добавить модель или прислать прогон

Прогнать модель и поделиться результатом может любой — это и есть «новые генерации». Модель —
это **факты в каталоге**, а канал доступа — адаптер.

Чтобы добавить **новую модель** в каталог:

1. Запись в `generation/models.yaml` (id, vendor, `access.adapter`, capabilities:
   `context_window`, `supports_seed`, `supports_tools`).
2. Числовые параметры в `generation/params.yaml` — по записи на модель (это проверяет
   `prism check`).
3. Если провайдер новый — один файл-адаптер в `harness/generate/adapters/` (есть готовые:
   OpenAI-совместимый, GigaChat, YandexGPT) плюс регистрация в `registry.py`.
4. API-ключ — в `.env` (в репозиторий ключи не идут).

> **Для категории B нужна модель с `supports_tools: true`** — там модель сама собирает
> метаданные базы вызовами инструментов. Без поддержки инструментов она не увидит схему и
> провалит ось Платформа. Инструменты сейчас поддерживают все модели каталога — и через
> OpenRouter (Claude/GPT/Gemini/DeepSeek…), и GigaChat, и YandexGPT. Флаг модели —
> `capabilities.supports_tools` в `generation/models.yaml`.

Прогон и шеринг результата:
```bash
uv run prism ping --models <ключ>                   # ключ рабочий и канал отвечает?
uv run prism generate --category A --models <ключ>  # сгенерировать код
uv run prism score                                  # оценить → results/auto/
uv run prism submit                                 # упаковать прогон (с compat_hash)
```
Готовый пакет из `results/submissions/` приложите к PR или пришлите мейнтейнеру — он сверит
совместимость версий (`prism submit --verify`) и вольёт результат в лидерборд.

## Правила оформления

- **Когда готово:** `prism check` и `make test` зелёные; перед коммитом — `make lint`.
- **Коммиты:** [Conventional Commits](https://www.conventionalcommits.org) — заголовок
  по-английски (проверяет `commitizen`). Документация и комментарии в коде — по-русски.
- **Один PR — одна логическая единица** (одна задача, одна модель, один фикс).
- **Не правьте `metrics/smop.yaml` без обсуждения** — это конституция метрики; изменение
  определений осей затрагивает все результаты. Сначала Issue.

## Уровень L2 — экспертная оценка (в планах)

L2 — это сверка машинной оценки с живым экспертом: специалист по 1С ставит SMOP-баллы вслепую
к авто-оценке, а мы считаем согласие машины и эксперта (каппа Коэна) — главный научный
результат проекта.

**Пока этого уровня по сути нет.** Есть только концепт-прототип платформы экспертной разметки —
[genlab-1c-web](https://github.com/4dand/genlab-1c-web); полноценная разметка с двумя-тремя
экспертами впереди. Хотите поучаствовать в L2 — напишите в Issue.
