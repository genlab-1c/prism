# Участие в PRISM

Спасибо, что хотите помочь! PRISM растёт силами сообщества 1С — и **самый ценный вклад
— это новые задачи**. Чем шире и честнее банк заданий, тем точнее бенчмарк отвечает на
вопрос «какая нейросеть лучше пишет код на 1С и в чём именно».

Этот документ — пошаговая карта: с чего начать, как добавить задачу или модель и как
убедиться, что вклад примут. Если хотите сначала понять, как устроена оценка, — загляните
в [«Как это работает»](docs/how-it-works.md) (методика простым языком).

## Дисциплина (почему PRISM можно доверять)

Три правила, нарушение любого — повод вернуть PR на доработку. Они же — суть проекта:

1. **Код читает данные, данные не живут в коде.** Пороги, банды, веса — только в
   `metrics/*.yaml`. Числа в документации не дублируются: проза ссылается на YAML.
2. **«Не измерено ≠ ноль».** Если ось нечем проверить — она честно исключается, а не
   занижает балл. Мы не публикуем метрику, которую инструмент не вправе утверждать.
3. **Эталон обязан проходить свои тесты на 100%.** Задача без рабочего `canonical.bsl` —
   это не задача, а гипотеза. `make check` это гейтит.

## С чего начать (любой контрибьютор)

```bash
git clone https://github.com/genlab-1c/prism && cd prism
make setup        # .venv + инструменты осей на хост (OneScript + BSL LS)
make check        # целостность: контракты метрики, задания, эталоны, инструменты
make test         # тесты харнесса (интеграционные сами скипнутся без инструментов)
```

Если `make check` зелёный — окружение готово. Перед коммитом ставится `pre-commit`
(хуки уже описаны в `.pre-commit-config.yaml`):

```bash
.venv/bin/pre-commit install
make lint         # ruff + pre-commit, как в CI
```

## Способы участвовать

| Вклад | Порог | Что нужно |
|---|---|---|
| **Задача категории A** (алгоритмика) | 🟢 низкий | только OneScript — ставится `make setup` |
| **Задача категории B** (платформа) | 🟠 выше | свой дистрибутив 1С + Docker-образ (см. ниже) |
| **Новая модель** в лидерборде | 🟢 низкий | запись в `generation/models.yaml` + свой API-ключ |
| **Улучшение харнесса** | 🟢–🟠 | Python + `make test` зелёный |
| **Экспертная оценка (L2)** | — | отдельный репозиторий [expert-platform](#уровень-l2--экспертная-платформа) |
| **Баг / идея задачи** | 🟢 | Issue на GitHub |

Идеи для первых задач и дорожная карта — в [`docs/status.md`](docs/status.md) и в
открытых Issues (ярлык `good first task`).

## 🟢 Добавить задачу категории A (рекомендуемый вход)

Категория A — алгоритмика на чистом BSL, исполняется в **OneScript** (бесплатно, ставится
`make setup`, не требует платформы 1С). Это самый простой способ войти — ~10 минут.

Задача — это каталог `tasks/category_a/<id>_<slug>/` из **трёх файлов**. Скопируйте
`tasks/category_a/A1_bubble_sort/` как образец и заполните:

**`task.yaml`** — условие:
```yaml
id: A99                     # свободный id (проверьте, что не занят в tasks/category_a/)
name: Краткое имя задачи
category: A
difficulty: easy            # easy | medium | hard
entry_point: ИмяФункции     # функция, которую вызовет харнесс
signature: "Функция ИмяФункции(Аргумент) Экспорт"
prompt: |
  Понятная формулировка для модели: что на входе, что вернуть,
  граничные условия. Без подсказок к реализации.
tags:
  skill: [коллекции, алгоритм]   # только значения из tasks/tags.yaml
```

**`tests.yaml`** — скрытые тест-кейсы (ось M):
```yaml
entry_point_patterns:       # регэкспы поиска функции в коде кандидата
  - "ИмяФунк\\w*"
tests:
  - args: [[3, 1, 2]]       # аргументы вызова
    expected: [1, 2, 3]     # ожидаемый результат (deep-equal)
  - args: [[]]
    expected: []
```
Кладите **граничные случаи** (пусто, один элемент, дубли, отрицательные, переполнение) —
именно на них модели и спотыкаются, ради этого тесты и скрытые.

**`canonical.bsl`** — эталонная реализация. Обязана проходить **все** свои тесты на 100% —
иначе задача не пройдёт `make check`.

Проверка вклада — одна команда:
```bash
make check        # ваш эталон прогоняется против ваших тестов
```
Зелёный `make check` = вашу задачу можно слать в PR. Это и есть «зелёный свет» контрибьютора.

> **Теги.** Берите только значения из `tasks/tags.yaml` (словарь закрыт — опечатка ловится
> `make check`). Если нужного навыка нет — предложите его в PR отдельным изменением словаря
> с обоснованием.

## 🟠 Добавить задачу категории B (платформенная)

Категория B — то, ради чего PRISM существует: код исполняется в **реальном 1С против
синтетической базы**, собираемой из описания задачи. Порог выше — нужен Docker-образ с
платформой.

### Барьер: образ 1С (принеси свой дистрибутив)

Дистрибутив платформы 1С **не распространяется** в репозитории по лицензии. Учебную версию
можно бесплатно получить на [developer.1c.ru](https://developer.1c.ru), затем собрать образ:

```bash
# положите дистрибутив рядом с docker/onec.Dockerfile, затем:
make setup-docker      # харнесс + docker-образы инструментов (режим MODE=docker)
```

**Лицензионное правило (жёсткое).** В git идут только **описания и скрипты сборки** — всё
собственного авторства. **Никогда не коммитьте**: дистрибутив платформы 1С, собранные
`.cf`/`.dt`, саму базу. База целиком воссоздаётся из `config_spec.yaml` + `fixtures.yaml`
при сборке.

### Анатомия B-задачи

Каталог `tasks/category_b/<id>_<slug>/` из **пяти файлов**. Образец для копирования —
`tasks/category_b/B1_stock_balances/`:

| Файл | Что описывает |
|---|---|
| `task.yaml` | условие, `entry_point_patterns`, `expected_objects` (объекты метаданных — сид оси P), теги |
| `config_spec.yaml` | **синтетическая** конфигурация: справочники, документы, регистры (НЕ копия УТ/БП) |
| `fixtures.yaml` | начальные данные базы (детерминированные) |
| `tests.bsl` | скрытые проверки на BSL, исполняются в 1С |
| `canonical.bsl` | эталон — обязан проходить `tests.bsl` на 100% |

Один прогон даёт сразу два сигнала: **M** (вернул правильный ответ) и **P** (не упал на
обращении к метаданным). Поэтому тесты пишите так, чтобы заведомо сломанный кандидат
(обращение к несуществующему регистру/реквизиту) **проваливался**, а не давал ложный зачёт.
Это типовая ошибка авторов задач — ловите её при написании эталона-«вредителя».

### CI не проверяет B автоматически

В CI нет платформы 1С → канонические B-эталоны там не валидируются. Поэтому в PR с B-задачей:

1. Прогоните локально `make check MODE=docker`.
2. **Приложите вывод** в описание PR (что эталон прошёл свои тесты).
3. Мейнтейнер с образом перепроверит.

## 🟢 Добавить модель в лидерборд

Модель — это **факты в каталоге**, а канал доступа — адаптер. Чтобы добавить модель:

1. Запись в `generation/models.yaml` (id, vendor, `access.adapter`, capabilities:
   `context_window`, `supports_seed`, `supports_tools`).
2. Числовые параметры в `generation/params.yaml` (по записи на модель — это гейтит `make check`).
3. Если провайдер новый — один файл-адаптер в `harness/generate/adapters/`
   (есть готовые: OpenAI-совместимый, GigaChat, YandexGPT) + регистрация в `registry.py`.
4. API-ключ — в переменную окружения (в репозиторий ключи не идут).

Прогон: `prism generate --category A --models <ключ>` → `prism score`.

## Правила оформления

- **Definition of done:** `make check` и `make test` зелёные; перед коммитом — `make lint`.
- **Коммиты:** [Conventional Commits](https://www.conventionalcommits.org) (заголовок —
  по-английски, проверяется `commitizen`). Документация и комментарии в коде — **по-русски**.
- **Один PR — одна логическая единица** (одна задача, одна модель, один фикс).
- **Не правьте `metrics/smop.yaml` без обсуждения** — это конституция метрики; изменение
  определений осей затрагивает все результаты. Сначала Issue.

## Уровень L2 — экспертная платформа

Экспертная разметка (живой человек ставит SMOP-баллы вслепую к авто-оценке) живёт в
**отдельном репозитории** — веб-платформе [expert-platform](https://github.com/genlab-1c/expert-platform).
Она подключает PRISM как submodule, берёт задачи и сгенерированный код, а оценки экспортирует
обратно файлом-контрактом. Ключевой научный результат — **согласие L1↔L2 (каппа Коэна)**.
Хотите оценивать как эксперт — заходите в тот репозиторий.

Любой вклад, даже одна задача категории A, делает ответ на «какая ИИ лучше для 1С» честнее.
