**PRISM** — открытый бенчмарк, который измеряет, **насколько хорошо ИИ пишет код для
1С:Предприятие 8**. Вместо одной оценки «зачёт/незачёт» он раскладывает качество
сгенерированного нейросетью BSL-кода на четыре оси (как призма раскладывает свет в
спектр) и **проверяет код исполнением** — компилятором, тестами и запуском против конфигурации.

Если вы ищете, **какая нейросеть лучше пишет код на 1С** (Claude, GPT, Gemini, YandexGPT,
GigaChat), занимаетесь **вайбкодингом 1С** или исследуете применимость **ИИ для генерации
кода 1С** — это инструмент, который даёт воспроизводимый, проверяемый ответ, а не
впечатление.

**Что умеет:**

- **Адаптеры к API моделей** — OpenRouter (Claude, GPT, Gemini, DeepSeek, Qwen, GLM…),
  GigaChat, YandexGPT, любой OpenAI-совместимый (Ollama/vLLM локально); вендор и канал
  доступа разделены — новый провайдер = один файл-адаптер.
- **По-настоящему исполняет код** — OneScript для алгоритмики (A) и headless-1С в Docker
  против синтетической базы для платформенных задач (B).
- **Сухой прогон без сети** (`--mock`) — прогнать весь путь генерация→оценка без ключей и денег.
- **Учёт стоимости** — предполётная смета (`--dry-run`), живой счётчик и кап бюджета
  (`--max-cost`) по датированной таблице цен.
- **Прокси на канал** (`PRISM_PROXY_RU`/`INTL`) — отдельный для отечественных и зарубежных провайдеров.
- **Надёжность больших прогонов** — чекпойнт и `--resume`, ретраи при сбоях сети, параллелизм.
- **Готовность и связь** — `prism doctor` (инструменты + ключи) и `prism ping` (живая проверка моделей).
- **Шеринг результатов** — `prism submit` пакует прогон с хешем совместимости версии бенчмарка.
- **Срезы по навыкам** — задачи помечены тегами (строки, даты, запрос, метаданные…): видно,
  с чем модель справляется хуже.

## Зачем это нужно

Популярные бенчмарки кодогенерации (HumanEval, MBPP) для 1С **не работают**:
встроенный язык 1С (BSL) завязан на метаданные конфигурации, русский синтаксис и
платформенные конструкции (регистры, виртуальные таблицы, срезы). Модель может
написать «красивый» код, который компилируется и выглядит верно, но **падает на
первом же реальном запросе к справочнику или регистру**. Обычный `pass/fail` этого
не ловит.

PRISM создан, чтобы дать индустрии 1С честный, открытый ответ на вопросы:
- какая ИИ-модель лучше генерирует код 1С и **в чём именно** (запросы, даты, метаданные…);
- можно ли доверять **вайбкодингу** в 1С и где он ломается;
- как объективно сравнивать LLM для доменно-специфичной платформы.

## SMOP — четыре оси качества

Сгенерированный код мы оцениваем не одной галочкой, а по четырём осям качества сразу —
как призма раскладывает свет в спектр. Главный результат — **вектор** `[S, M, O, P]`, а не
одно число: одно число прячет, *где именно* модель ошиблась.

| Ось качества | Что измеряет | Чем проверяется |
|---|---|---|
| **S** — Syntax (Синтаксис) | компилируется ли модуль | парсер BSL Language Server / компилятор 1С |
| **M** — Meaning (Семантика) | делает ли код то, что просили | исполнение скрытых тестов |
| **O** — Optimization (Оптимальность) | оптимально ли решение (класс алгоритма, антипаттерны) | исполнение на датасете входных значений (A) + анализ кода (B) |
| **P** — Platform (Платформа) | корректно ли используются метаданные платформы | исполнение против синтетической базы |

## Как устроено

> Методика оценки — **[Как это работает](docs/how-it-works.md)**.

- **Два уровня оценки.** L1 — авто-оракулы (компилятор, тесты, исполнение): быстро и
  воспроизводимо. L2 — экспертная разметка: калибрует L1. Ключевой научный результат —
  согласие L1 с экспертом.
- **Две категории задач.** **A** — алгоритмика на чистом BSL (исполняется в OneScript).
  **B** — платформенные задачи (FIFO, остатки, цены, себестоимость, валюты), которые
  **исполняются в headless-1С против синтетической базы**, собираемой из описания задачи.
- **Срезы по тегам.** Каждая задача помечена навыками (`строки`, `даты`, `запрос`,
  `метаданные`…) — отчёт показывает, **с чем модель справляется лучше, а с чем хуже**.
- **Правило «не измерено ≠ ноль».** Нет инструмента оси — ось честно исключается, а не
  занижает оценку.

## Быстрый старт

Два инструмента: **`make`** — *поставить* (окружение + инструменты осей), **`prism`** —
*пользоваться* (своя справка и флаги). Окружение управляет
[uv](https://docs.astral.sh/uv/); нет uv? — `curl -LsSf https://astral.sh/uv/install.sh | sh`.

```bash
make setup           # один раз: окружение (uv sync) + инструменты осей на хост (категория A)
make setup-all       # то же + учебная 1С для категории B (проведёт по дистрибутиву)
source .venv/bin/activate   # активировать — дальше просто `prism ...`

prism                # шпаргалка: что делать
prism doctor         # всё ли установлено: инструменты осей и ключи моделей
prism leaderboard    # посмотреть результаты (мгновенно, без пересчёта)
prism check          # целостность: контракты, задания, эталоны, инструменты
prism score          # пересчитать оценку L1
```

Не активируя `.venv`, любую команду можно звать через `uv run prism <команда>`. Все флаги —
`prism <команда> --help`.

Категория B исполняется в Docker против учебной 1С (дистрибутив приносится свой,
в репозиторий не попадает): `make setup-all` или `make image-onec` проведёт по шагам —
см. [`docker/onec.Dockerfile`](docker/onec.Dockerfile). Проверить готовность — `prism doctor`.
Подробности — [документация](#документация).

## Свой прогон: генерация → оценка

Потыкаться руками — два шага: модели пишут код, потом считаем баллы.

```bash
# 1. ключи моделей (см. .env.example): OPENROUTER_API_KEY для Claude/GPT/Gemini,
#    GIGACHAT_AUTH_KEY для GigaChat, YANDEX_API_KEY/FOLDER_ID для YandexGPT
cp .env.example .env            # впишите свои ключи — .env подхватится автоматически
prism doctor                    # всё ли готово (инструменты + ключи); prism ping — связь с моделями

# 2. генерация: модели решают задачи → results/experiment_*.json
prism generate --category A --mock                     # сухой прогон без сети (попробовать конвейер)
prism generate --category A --models claude            # одна модель, все задачи A
prism generate --category B --models gigachat gpt      # несколько моделей сразу

# 3. оценка готовых генераций по осям S·M·O·P
prism score                                            # свежий прогон A и B → лидерборды
prism submit                                           # упаковать результат, чтобы поделиться
```

Сегодня GigaChat, завтра Claude — в **тот же** прогон через `--resume` (дозапишет только
недостающие пары):

```bash
prism generate --category A --models claude --resume experiment_A_<timestamp>
```

Модели берутся из [`generation/models.yaml`](generation/models.yaml) (GigaChat/YandexGPT
там по умолчанию закомментированы — раскомментируйте нужные). Все флаги — в [CLI](docs/cli.md).

## Производительность и железо

Прогоны кандидатов идут **параллельно** (env `PRISM_CONCURRENCY`, по умолчанию ≤4) — на
баллы это не влияет (проверено бит-в-бит), только на скорость. Категория A лёгкая
(OneScript, доли секунды на кандидата); категория B тяжелее: каждый прогон поднимает
headless-1С в Docker — на практике **секунды, не минуты** (~10–20 с на кандидата, пустая
конфигурация кэшируется), а параллелизм масштабирует по ядрам (напр. 75 B-кандидатов ≈
5–6 мин в 3 потока).

Грубо по ресурсам:

| | Категория A | Категория B |
|---|---|---|
| Чем считается | OneScript + BSL LS (Java) | headless-1С в Docker |
| CPU | 2+ ядра | 4+ ядра (≈ по ядру на параллельный прогон) |
| RAM | ~2 ГБ | ~2 ГБ на один параллельный прогон 1С (4 потока ≈ 8–12 ГБ) |
| Прочее | — | Docker + образ `prism-onec` |

Не хватает памяти — снизьте `PRISM_CONCURRENCY` или задайте лимиты `PRISM_ONEC_MEMORY` /
`PRISM_ONEC_CPUS`. При нехватке прогон честно помечается «не измерено», а не выдаёт неверный балл.

<sub>⚠️ <b>O-исп</b> (категория A) — оптимальность здесь мерится <b>исполнением</b>: решение
гоняется на входе растущего размера, число выполненных операций (oscript <code>-codestat</code>)
даёт класс роста, который сравнивается с оптимальным для задачи. Так машина различает быстрый
алгоритм и медленный — перебор парами вместо словаря, склейка строк в цикле, запрос в цикле
учитываются подменой встроенных команд на инструментированные аналоги. В категории <b>B</b>
оптимальность пока статическая (<b>O-авто</b> — список perf-антипаттернов BSL LS). Неалгоритмическую
оптимальность (выбор структуры, стиль) машина не судит — это Уровень 2 (эксперт).</sub>

> **Что показал прогон:** на категории A **DeepSeek V4-Flash** уверенно ведёт и по смыслу
> (M), и по оптимальности (O) при копеечной цене. Ось **O** меряется исполнением и честно
> ставит **N/A**, когда код не запускается: например, YandexGPT 5.1 Pro пишет настоящие
> алгоритмы, но с 1С-ошибками — код не исполняется, и оптимальность не оценивается (а не
> получает дармовую десятку). В категории B ось **P** различает, выдумывает ли модель
> метаданные: у DeepSeek код переживает контакт с базой (P=4.7), у Gemini — нет (P=0).
>
> Главное — **вектор**, а не одно число; **Q** вторичен (грубое ранжирование). Это первый
> авто-прогон Уровня 1 (одна генерация на задачу, дешёвые модели) — границы применимости в
> [`docs/validity.md`](docs/validity.md).

## Структура репозитория

```
metrics/     правила метрики SMOP — как из запуска кода получается балл по каждой оси
             (только настройки в YAML, без кода)
tasks/       сами задания: условие + скрытые тесты + эталонное решение.
             category_a/ — алгоритмика (чистый 1С) · category_b/ — платформенные (запуск в 1С)
harness/     код бенчмарка: сгенерировать код моделями, оценить по осям S·M·O·P,
             запустить в OneScript и headless-1С, посчитать статистику
generation/  с кем и как работаем: список моделей, их параметры, системные промпты
results/     что получилось: сырые ответы моделей и автоматические оценки
docs/        документация (та, что ты сейчас читаешь)
```

## Статус

Активная разработка (alpha). Категория A (оси S/M/O) и **исполнение категории B**
(оси M/P против синтетической базы) — рабочие, эталоны гейтятся в `prism check`.
Дорожная карта и честные ограничения — [`docs/status.md`](docs/status.md).

## Участие

PRISM растёт силами сообщества 1С, и самый ценный вклад — **новые задачи**. Низкопороговый
вход — задача категории A (только OneScript, ~10 минут): см. [`CONTRIBUTING.md`](CONTRIBUTING.md).

## Благодарности

PRISM стоит на плечах open-source инструментов сообщества 1С — без них исполняемой
оценки не было бы:

- **[BSL Language Server](https://github.com/1c-syntax/bsl-language-server)** (проект
  `1c-syntax`) — статический анализ кода для осей **S** и **O**.
- **[OneScript](https://oscript.io/)** — движок языка 1С, на котором исполняются задачи
  категории A (ось **M**).
- **[YAXUNIT](https://github.com/firstBitMarksistskaya/yaxunit)** — модульное тестирование
  в 1С (целевой формат проверок категории B).
- **[1C-MCP](https://github.com/vladimir-kharin/1c_mcp)** (Владимир Харин) — MCP-серверы
  в 1С: доступ ИИ к метаданным конфигурации, на котором держатся контекст-режим и будущее
  агентное издание.
- **Сообщество 1С и [Infostart](https://infostart.ru/)** — за экспертизу, обратную связь
  и культуру, в которой такой инструмент вообще имеет смысл.

Отдельная признательность всем, кто присылает **новые задачи** — именно из них и растёт
ценность бенчмарка.

## Происхождение

PRISM вырос из научной работы «Экспериментальная оценка эффективности ИИ в генерации
кода для доменно-специфичных платформ (на примере 1С:Предприятие 8)» (Андреев Д.С.,
март 2026) — [репозиторий статьи](https://github.com/4dand/1c-ai-codegen-research-paper).
Концепт методики SMOP оттуда переработан в исполняемый бенчмарк.
